# 教师主页功能回归 Spec

## 1. 需求重写

### 1.1 背景

旧工作区曾实现过“教师主页 / 导师名录”功能，但因版权和呈现风格问题未合并。现在协会允许该页面上线，但需要重新设计前端，并补齐数据同步机制。

当前目标不是把旧页面原样迁回，而是在正式前后端仓库中实现一个克制、可信、可维护的教师信息页面。

### 1.2 功能目标

新增一个公开访问的教师主页功能，用于展示协会当前重点对接教师。首批上线名单为：

- 李松
- 杜文亮
- 林峰
- 申大坤

产品边界：

- 首批正式上线只公开李松、杜文亮、林峰、申大坤四位教师。
- 后台保留维护名单的能力，用于后续替换或新增重点对接教师。
- 新增、替换或停用教师属于内容发布动作，必须经过后台审校后才进入前台，不自动扩展为全院教师列表。

页面需要满足：

- 信息来源以浙江大学网络空间安全学院教师队伍页面和可访问的教师个人主页为准。
- 前端视觉风格严肃、朴素、大气，与官网现有导航、页脚、浅色 / 深色主题变量保持一致。
- 不复用旧页面的粒子背景、霓虹发光、雷达图、HUD、终端风格、复杂动效等强赛博视觉。
- 页面可用于学生快速了解教师方向、联系方式和外部主页入口。
- 不开放全院教师列表，也不保留旧数据中的其他教师。

### 1.3 数据同步需求

“实时同步”定义为后端缓存式同步，而不是前端每次访问直接爬取源站。

后端需要提供教师信息公开 API。API 返回的数据由同步器从源站抓取并写入本地缓存 / 数据库。前端只消费本站 API，不直接访问学院站点或教师个人主页。

同步策略建议：

- 支持手动触发同步，便于上线前和内容变更后立即刷新。
- 支持定时或 TTL 同步，建议最小粒度为 12-24 小时。
- 同步失败时保留最近一次成功数据，不因源站暂时不可用导致前端空白。
- 公开 API 只返回前台展示需要的字段，例如教师资料、外部主页、主要来源和 `last_synced_at`；`source_status`、`sync_note` 等同步诊断字段只允许出现在后台接口。
- 源站字段缺失或个人主页暂未开放时，允许使用学院教师队伍页中的基础信息和人工维护的补充链接。

### 1.4 数据字段范围

教师信息以“事实型字段”为主，避免营销化、夸张化、未经来源验证的表述。

每位教师建议字段：

- `id`：稳定英文标识，例如 `li-song`、`du-wenliang`。
- `name`：中文姓名。
- `title`：职称 / 岗位头衔。
- `affiliation`：所属学院或单位。
- `research_areas`：研究方向列表。
- `bio`：简短介绍，优先来自公开主页；如需人工整理，文案应客观克制。
- `email`：公开邮箱；源站未提供时为空。
- `homepage_urls`：学院页、浙大个人主页、自建主页等外部链接。
- `avatar_url`：头像地址；优先本地缓存后的图片或明确可用的公开头像。
- `source_url`：主要数据来源。
- `last_synced_at`：本地最后成功同步时间。
- `profile_sections`：可选的人工审校补充内容，用于后续展示扩展信息。

后台维护记录可以额外保存 `source_status`、`sync_note`、`source_config`、`field_sources`、`draft` 等内部字段，但这些字段不得从公开 API 泄露到前台。

旧文件 `old/csa-website-gitee/src/assets/data/professors.jsonc` 只作为历史参考：

- 可参考四位教师的旧字段组织方式、头像路径和部分候选链接。
- 不直接复制旧文案作为最终展示内容。
- 不迁移旧数据中的其他教师。
- 不迁移旧页面中不存在或无法确认版权 / 来源的 gallery 图片。

### 1.5 前端页面范围

新增公开路由，建议路径为 `/roster` 或 `/mentors`，最终路径需在实现前确认。

具体视觉和信息组织应遵循 `docs/plan/teacher-roster-design-guide.md`。该设计指引优先保证页面自然融入现有 CSA 官网，不直接套用任何单一外部主题。

页面结构建议：

- 顶部介绍区：说明页面用途，强调“协会教师资源 / 科研方向对接”，不做营销型大 hero。
- 教师列表区：教师卡片，展示头像、姓名、职称、研究方向标签、联系方式入口。
- 详情区：点击卡片后进入详情页，或在同页展开教师详细信息。若实现详情页，建议路径为 `/roster/:id`。
- 来源说明区：展示数据来源和最后更新时间，避免用户误以为所有内容由协会人工实时确认。
- 空状态 / 错误状态：API 失败时显示温和提示，并避免页面布局崩溃。

视觉要求：

- 使用现有 CSS 变量：`--bg-secondary`、`--bg-surface`、`--text-primary`、`--text-secondary`、`--accent-color`、`--border-color` 等。
- 支持浅色和深色主题。
- 卡片圆角、阴影、边框强度与现有新闻 / 活动 / 关于页面协调。
- 保留轻量 hover 和过渡即可，不使用大面积动画或特效背景。
- 移动端优先保证可读性和信息密度，不出现文字溢出、按钮挤压、头像变形。

### 1.6 后端页面范围

新增公开教师 API，建议：

- `GET /api/teachers/list`：返回当前公开教师列表和公开更新时间。
- `GET /api/teachers/detail?id=...`：返回单个教师详情。
- `POST /api/teachers/sync`：管理员或发布者权限下手动触发同步。

后端同步器建议分层：

- source fetcher：负责请求标准浙大个人主页；学院教师队伍页和非标准个人主页作为人工维护来源。
- parser：负责从 HTML 中抽取姓名、职称、研究方向、邮箱、头像、更新时间等字段。
- normalizer：负责只发布后台启用且已审校的教师记录，并统一字段格式。
- storage：负责写入数据库或缓存文件。
- fallback：负责源站失败时返回最后一次成功数据。

如果第一阶段预算有限，可以先实现静态 seed + API，再在第二阶段替换为同步器，但前端接口形态应保持不变。

### 1.7 非目标

本次不做：

- 全院教师数据库或自动导入全院教师列表。
- 招生状态判断，例如“招人中 / 已满”。
- 论文列表自动同步。
- 教师成果排行、雷达能力图、评分、推荐算法。
- 复杂图片画廊。
- 跳过审核直接发布同步内容。
- 从旧 Gitee 工作区继续开发。

## 2. 验收标准

### 2.1 内容验收

- 首批上线页面只出现李松、杜文亮、林峰、申大坤四位教师；后续扩展名单必须经过后台审校。
- 每位教师至少展示姓名、职称、研究方向、外部主页入口和数据来源。
- 能清楚看到数据最后同步时间。
- 旧 `professors.jsonc` 中的其他教师不会进入正式前端产物。
- 页面文案客观，不出现夸张宣传语、未经来源确认的奖项或项目表述。
- 对个人主页暂未开放或字段缺失的教师，页面应显示可理解的缺省状态，而不是编造信息。

### 2.2 同步验收

- 后端公开 API 可在无登录状态下读取教师列表和详情。
- 手动同步接口仅管理员或具备发布 / 管理权限的账号可调用。
- 同步成功后，API 返回更新后的教师数据和 `last_synced_at`。
- 同步失败时，公开 API 仍返回最近一次已发布资料；失败状态或说明只在后台同步草稿中展示。
- 源站 HTML 结构变化导致字段缺失时，同步器不会抛出未捕获异常。
- 同步器只处理后台配置的教师记录，不因源站页面新增教师而自动扩展展示范围。

### 2.3 前端验收

- 新页面可从导航栏或合适入口访问，且刷新后路由正常。
- 浅色和深色主题下均无明显低对比度文本。
- 桌面端四位教师展示稳定，布局不抖动。
- 移动端 375px 宽度下无横向滚动，无文字严重溢出，无按钮遮挡。
- API loading、empty、error 三种状态都有可用 UI。
- 外部链接在新窗口打开，并使用 `rel="noopener noreferrer"`。
- 头像缺失时显示一致的占位样式，而不是破图。

### 2.4 工程验收

- 前端通过 `pnpm build`。
- 后端启动后 `/docs` 中能看到教师相关 API。
- 新增后端逻辑有最小单元测试或可重复执行的同步验证脚本。
- 新文件开头包含职责范围和功能边界说明。
- 不修改旧快照目录 `old/csa-website-gitee/`。
- 不引入与该功能无关的大型依赖。
- 不把旧 `node_modules`、`dist`、临时爬取产物提交进正式仓库。

## 3. 多阶段实现计划

### 阶段 0：需求确认和源站字段勘探

目标：把不确定项降到实现可控范围。

工作项：

- 确认正式路由命名：`/roster`、`/mentors` 或其他。
- 确认页面入口放在导航栏、关于页、首页信息区，还是多处入口。
- 确认四位教师的权威来源优先级：学院教师队伍页、浙大个人主页、自建主页。
- 建立字段映射表，记录每位教师哪些字段可自动同步，哪些字段需要人工兜底。
- 明确“实时同步”的频率要求和部署环境是否支持定时任务。

交付物：

- 字段映射表。
- 最终接口字段草案。
- 确认后的页面路径和入口位置。

### 阶段 1：后端静态 API 和前端新页面骨架

目标：先让功能以稳定接口形态跑通，避免前端被爬虫不确定性阻塞。

工作项：

- 后端新增教师 API 路由，使用 seed 数据返回四位教师。
- 前端新增教师列表页和可选详情页。
- 接入现有主题变量、导航、页脚和响应式布局。
- 从旧快照中仅迁移明确可用且风险较低的头像资源；不迁移 gallery。
- 增加 loading、empty、error 状态。

交付物：

- 可访问的新教师页面。
- `GET /api/teachers/list` 和可选 `GET /api/teachers/detail`。
- 初版可审核 UI。

验收重点：

- 路由、布局、主题和移动端表现。
- 文案语气是否符合“严肃朴素”。
- 旧页面强特效完全未迁入。

### 阶段 2：同步器和缓存机制

目标：实现与源站的可控同步。

工作项：

- 对配置了标准浙大个人主页的教师编写 parser；不可访问或非标准主页时记录状态并转为人工维护。
- 设计缓存存储方式，优先复用现有数据库模式；若要快速落地，可先用结构化 JSON 缓存文件。
- 实现同步失败 fallback。
- 新增手动同步 API，并加权限保护。
- 增加同步日志或最小状态记录。

交付物：

- 可重复执行的同步脚本 / 服务函数。
- 管理权限下的手动同步接口。
- 后台 API 返回同步状态和更新时间；公开 API 只返回展示字段和公开更新时间。

验收重点：

- 源站可用时能刷新数据。
- 源站不可用时前端仍可展示缓存。
- 只同步后台配置的教师记录，不从源站自动扩展名单。

当前落地状态：

- 后端已采用结构化 JSON 缓存文件 `backend/data/teachers.json`，暂不引入数据库迁移。
- 普通公开 API 只读取本站缓存，不在用户访问时请求外部源站。
- `POST /api/teachers/sync` 已接入手动同步，并继续使用发布者权限保护。
- 同步器只对标准 `person.zju.edu.cn/<slug>` 教师个人主页做程序化同步；李松、申大坤当前由后台人工维护，杜文亮、林峰配置了标准个人主页。
- 同步结果先写入后台待审核草稿，管理员发布后才影响前台公开资料；公开 API 不返回 `source_status` 或 `sync_note`。
- 当前未实现后台定时任务和头像本地下载；上线前或内容变更后由发布者手动触发同步。
- 后端提供 `scripts/verify_teachers_json.py` 和 `scripts/verify_teachers_sync.py` 作为可重复验证入口。

### 阶段 3：内容审校和 UI 精修

目标：把页面从“能用”调整到“适合正式上线”。

工作项：

- 审核每位教师展示文案，删除无来源或过度宣传表达。
- 对头像、标签、外部链接和来源说明进行统一排版。
- 检查浅色 / 深色主题、桌面 / 移动端布局。
- 根据审核意见调整导航入口和页面命名。
- 补齐必要的无障碍属性，例如图片 `alt`、按钮名称、链接说明。

交付物：

- 上线候选 UI。
- 审核后的教师数据。
- 前端截图或本地预览记录。

验收重点：

- 页面气质与官网其他页面协调。
- 信息可信、克制、可追溯。
- 无破图、溢出、错链。

### 阶段 4：测试、文档和上线准备

目标：降低上线后维护成本。

工作项：

- 跑前端构建和后端启动检查。
- 对同步器做至少一次成功同步和一次失败 fallback 验证。
- 更新 README 或维护文档，说明同步方式、数据来源、手动刷新方法。
- 如果部署环境支持定时任务，配置同步频率；否则记录人工刷新流程。
- 分别检查 frontend / backend Git 状态，按独立仓库提交。

交付物：

- 通过构建的前端。
- 可启动的后端 API。
- 维护说明。
- 待提交变更清单。

验收重点：

- 后续维护者知道如何刷新数据和排查同步失败。
- 功能不依赖旧快照目录。
- 前后端仓库变更边界清晰。
